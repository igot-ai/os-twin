# deepagents-cli MCP Bug Report

## Summary

https://github.com/langchain-ai/deepagents/pull/2228

**deepagents-cli v0.0.34** cannot execute MCP tool calls. All MCP tools (memory, channel, warroom) fail with `ClosedResourceError` at runtime, even though tool discovery (listing) works fine.

This affects ALL MCP transports: stdio, SSE, and HTTP.

## Root Cause

In `server_graph.py` (generated at runtime by deepagents-cli), MCP tools are loaded at graph initialization:

```python
# server_graph.py line ~170
mcp_tools, _, mcp_server_info = asyncio.run(
    resolve_and_load_mcp_tools(
        explicit_config_path=config.mcp_config_path,
        no_mcp=config.no_mcp,
        trust_project_mcp=config.trust_project_mcp,
        project_context=project_context,
    )
)
```

The `_` discards the `MCPSessionManager` — the object that holds the persistent MCP sessions (stdio processes, SSE connections). When `asyncio.run()` completes, the event loop closes, and all sessions die.

Later, when the LLM agent calls an MCP tool, the tool tries to use the dead session → `ClosedResourceError`.

## Proof: The Bug Is in deepagents-cli

### 1. The MCP servers themselves are healthy

The exact same memory MCP server works perfectly with Codex and Claude Code:

```
# Codex connects to the SAME server and calls save_memory successfully
$ codex /mcp

  • Tools: find_memory, grep_memory, memory_tree, save_memory, search_memory
  
# ostwin mcp test confirms the server is healthy

memory     ✓ Connected   5 tools (v1.27.0)
warroom    ✓ Connected   3 tools (v1.27.0)
```

### 2. Tool LISTING works — tool CALLING fails

deepagents successfully connects and lists tools (inside `asyncio.run()`):

But when the agent later calls a tool, the request **never reaches the server**:

The daemon log shows zero `save_memory` requests — deepagents' internal session is dead before the HTTP request is sent.

### 3. ALL MCP tools fail — not just memory

Channel MCP also crashes with the same error (from `video-sharing-16` run):
```
# room-000/artifacts/architect-output.txt.wrapper.log
🔧 Calling tool: channel_read_messages
⚠ Remote connection lost (RemoteException), retrying in 2s (1/3)…
⚠ Remote connection lost (RemoteException), retrying in 4s (2/3)…

```

Channel/warroom "work" in ostwin only because the **manager loop uses PowerShell scripts** (`Post-Message.ps1`, `Read-Messages.ps1`) that write directly to `channel.jsonl` files — they bypass MCP entirely. The MCP channel server was never successfully called by deepagents.

### 4. The exact line that causes the bug

File: `/tmp/deepagents_server_*/server_graph.py` (generated by deepagents-cli at runtime)

```python
mcp_tools, _, mcp_server_info = asyncio.run(
    resolve_and_load_mcp_tools(...)
)
#           ^ THIS DISCARDS THE MCPSessionManager
```

The `MCPSessionManager` (returned as the 2nd element) holds the live SSE/stdio sessions. Assigning it to `_` means Python garbage-collects it, closing all connections. The `mcp_tools` list still has tool definitions but the underlying transport is dead.

### 5. The bug was already known

Comment in `Invoke-Agent.ps1` line 280:
```powershell
# If no_mcp is set in role config, skip MCP entirely to avoid ClosedResourceError
# on remote LangGraph execution. Role scripts handle channel communication instead.
```

This confirms the ostwin team already knew MCP doesn't work with deepagents and intentionally set `no_mcp: true` as a workaround.

### 6. Tested all transports — all fail identically

| Transport | Tool List | Tool Call | Error |
|-----------|-----------|-----------|-------|
| stdio     | ✅ Works  | ❌ Fails  | `ClosedResourceError` |
| SSE       | ✅ Works  | ❌ Fails  | `ClosedResourceError` |
| Both fail because `asyncio.run()` closes the event loop, killing all sessions regardless of transport.

## Affected Components

- **File**: `deepagents_cli/server_manager.py` → generates `server_graph.py`
- **File**: Generated `/tmp/deepagents_server_*/server_graph.py`
- **Function**: `_build_tools()` in `server_graph.py`
- **Version**: deepagents-cli 0.0.34, deepagents SDK 0.4.11
- **MCP SDK**: 1.27.0

## Why Other Clients Work

| Client | MCP Works? | Reason |
|--------|-----------|--------|
| **Codex (OpenAI)** | ✅ Yes | Keeps MCP stdio process alive for entire session |
| **Claude Code** | ✅ Yes | Keeps MCP stdio process alive for entire session |
| **deepagents-cli** | ❌ No | Loads tools via `asyncio.run()`, discards session manager, session dies |

## Proposed Fix (in deepagents-cli)

The `server_graph.py` should keep the `MCPSessionManager` alive and pass it to the graph execution context:

```python
# CURRENT (broken):
mcp_tools, _, mcp_server_info = asyncio.run(
    resolve_and_load_mcp_tools(...)
)

# PROPOSED FIX:
# Keep session manager alive for the lifetime of the server
_mcp_session_manager = None

async def _load_mcp():
    global _mcp_session_manager
    tools, manager, info = await resolve_and_load_mcp_tools(...)
    _mcp_session_manager = manager  # Keep alive!
    return tools, manager, info

mcp_tools, _, mcp_server_info = asyncio.run(_load_mcp())

# OR better: don't use asyncio.run() — integrate into the server's event loop
```

Alternatively, the MCP session initialization should happen inside the LangGraph server's async lifespan, not at module import time.

## Workarounds

### 1. Shell CLI (current workaround)
Set `no_mcp: true` for all roles. Agents call memory via shell tool:
```bash
python3 ~/os-twin/A-mem-sys/memory-cli.py save "content" --name "name" --path "path" --tags "tags"
```

**Pros**: Works now, no deepagents changes needed
**Cons**: Agents must be prompted to use shell commands, LLM may not follow instructions reliably

### 2. Monkey-patch server_graph.py (TESTED — DOES NOT WORK)
Changed `_` to `_mcp_session_manager` in `server_graph.py` to keep the session manager alive.

**Result**: Session manager object stays alive, but the async sessions inside it are bound to the `asyncio.run()` event loop which is destroyed when `asyncio.run()` returns. The SSE/stdio connections are closed regardless.

**Conclusion**: The fix must be architectural — MCP sessions need to live in the LangGraph server's event loop, not a throwaway `asyncio.run()` loop. This requires changes inside deepagents-cli.

### 3. ACP mode (deepagents --acp)
ACP mode uses `await resolve_and_load_mcp_tools()` inside the main event loop, keeping sessions alive. This is the correct architecture. However, `--acp` runs as a server (not one-shot agent), so it doesn't map directly to ostwin's agent execution model.

### 3. SSE Daemon (tested, blocked by this bug)
Run memory MCP server as persistent HTTP daemon on localhost:6463. deepagents connects via SSE.

**Pros**: Server stays alive, correct architecture
**Cons**: Still fails because deepagents discards the SSE session too (same bug)

## Steps to Reproduce

1. Create an MCP config with any stdio or SSE server
2. Set `no_mcp: false` in agent config
3. Run `ostwin run <plan>` or `deepagents` CLI with `--mcp-config <path>`
4. Agent lists tools successfully
5. Agent calls any MCP tool → `ClosedResourceError`
