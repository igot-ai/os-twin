# Plans, Epics, and the DAG

## The Hierarchy

```
Plan
 |
 +-- PLAN-REVIEW  (always room-000, architect reviews before any epic starts)
 |
 +-- EPIC-001 ──depends_on──> EPIC-002
 |                              |
 +-- EPIC-003 ──depends_on──> EPIC-001, EPIC-002
 |
 +-- EPIC-004 ──depends_on──> EPIC-003
```

A **Plan** is the top-level unit of work. It contains multiple **Epics** linked
by a dependency DAG. Each epic maps 1:1 to a **War-Room** at execution time.

## Plan Definition

Plans are Markdown files stored in `.agents/plans/`. The format:

```markdown
# Plan: Rebuild Authentication

> Created: 2026-04-01T10:00:00Z
> Status: draft

## Config
working_dir: ./auth-service

## Goal
Replace the legacy auth system with JWT-based authentication.

## Epics

### EPIC-001 -- Design Auth Architecture
Roles: architect
Objective: Produce ADRs and component diagrams

#### Definition of Done
- [ ] ADR for JWT vs session tokens written
- [ ] Component diagram covers all auth flows
- [ ] Threat model documented

#### Acceptance Criteria
- [ ] ADR reviewed and approved by team lead
- [ ] Diagram includes OAuth2 and API-key flows

depends_on: []

### EPIC-002 -- Implement Core Auth
Roles: engineer, qa
Objective: Build the JWT auth module

#### Definition of Done
- [ ] Login/logout endpoints working
- [ ] Token refresh implemented
- [ ] Unit test coverage >= 80%

#### Acceptance Criteria
- [ ] Valid JWT returns 200
- [ ] Expired JWT returns 401
- [ ] Refresh token rotates correctly

depends_on: ["EPIC-001"]
```

### Per-Plan Files

When a plan is created, several files are generated:

| File | Purpose |
|------|---------|
| `{plan_id}.md` | The plan markdown content |
| `{plan_id}.meta.json` | Metadata: plan_id, title, working_dir, warrooms_dir, status, launched_at |
| `{plan_id}.roles.json` | Per-role overrides for this plan (model, temperature, skill_refs) |
| `{plan_id}.refined.md` | AI-expanded version (generated by `Expand-Plan.ps1`) |

## Epic Structure: The Three Components

Every epic has three mandatory sections:

### 1. Definition of Done (DoD)

The objective checklist. These are the conditions that must be true for the
epic to be considered complete. They are stored in `config.json` under
`goals.definition_of_done` as an array of strings.

```markdown
#### Definition of Done
- [ ] Core logic implemented
- [ ] Unit test coverage >= 80%
- [ ] Lint clean
- [ ] Documentation updated
```

### 2. Acceptance Criteria (AC)

The behavioral verification. These describe how the system should behave when
the epic is done. Stored in `config.json` under `goals.acceptance_criteria`.

```markdown
#### Acceptance Criteria
- [ ] User can login with valid JWT
- [ ] System rejects expired tokens with 401
- [ ] Rate limiting is active on auth endpoints
```

### 3. Tasks

Tasks are the work breakdown within an epic. They live in `TASKS.md` inside
the war-room directory and are tracked by agents during execution:

```markdown
- [x] TASK-001 -- Design and plan implementation
  - AC: All requirements documented
  - AC: Architecture diagram complete
- [ ] TASK-002 -- Implement core logic
  - AC: All unit tests pass
  - AC: No lint warnings
- [ ] TASK-003 -- Write integration tests
  - AC: Coverage report generated
```

## Epic Metadata Directives

Each epic section can carry inline directives parsed from its markdown:

| Directive | Example | Purpose |
|-----------|---------|---------|
| `Roles:` | `Roles: engineer, qa` | Which agent roles work on this epic |
| `Objective:` | `Objective: Build the auth module` | Mission directive for the agent |
| `Working_dir:` | `Working_dir: ./auth` | Override working directory |
| `Pipeline:` | `Pipeline: standard` | Pipeline variant |
| `Capabilities:` | `Capabilities: code, test` | Required agent capabilities |
| `depends_on:` | `depends_on: ["EPIC-001"]` | Upstream dependencies |

## The DAG (Directed Acyclic Graph)

### How It Works

1. `Start-Plan.ps1` parses the plan markdown and extracts `depends_on` from each epic
2. Every epic automatically gets `PLAN-REVIEW` injected as a dependency
3. `Build-DependencyGraph.ps1` runs **Kahn's algorithm** for topological sorting
4. The result is `DAG.json` in the war-rooms directory

### DAG.json Structure

```json
{
  "generated_at": "2026-04-07T09:09:52Z",
  "total_nodes": 7,
  "max_depth": 5,
  "nodes": {
    "PLAN-REVIEW": {
      "room_id": "room-000",
      "depends_on": [],
      "dependents": ["EPIC-001", "EPIC-002"],
      "depth": 0,
      "on_critical_path": true
    },
    "EPIC-001": {
      "room_id": "room-001",
      "depends_on": ["PLAN-REVIEW"],
      "dependents": ["EPIC-002"],
      "depth": 1,
      "on_critical_path": true
    }
  },
  "topological_order": ["PLAN-REVIEW", "EPIC-001", "EPIC-002"],
  "critical_path": ["PLAN-REVIEW", "EPIC-001", "EPIC-002"],
  "critical_path_length": 3,
  "waves": {
    "0": ["PLAN-REVIEW"],
    "1": ["EPIC-001"],
    "2": ["EPIC-002"]
  }
}
```

### Key DAG Concepts

| Concept | Meaning |
|---------|---------|
| **PLAN-REVIEW** | Universal root node (room-000). An architect reviews the plan before any epic starts. |
| **depends_on** | Upstream epics that must reach `passed` before this epic can begin |
| **dependents** | Downstream epics waiting on this one |
| **depth** | Distance from root. Determines the wave. |
| **waves** | Epics at the same depth can execute in parallel |
| **critical_path** | Longest dependency chain. Delays here delay the entire plan. |
| **topological_order** | Valid sequential execution order respecting all dependencies |

### Dependency Gating

The manager loop checks dependencies before spawning a worker:

1. Read `DAG.json` to find this epic's `depends_on` list
2. For each dependency, check the corresponding room's `status` file
3. If all dependencies are `passed`, the epic is ready to start
4. Otherwise, the epic stays `pending`

This is implemented in `.agents/plan/Test-DependenciesReady.ps1`.

### Two-Stage DAG Generation

| Stage | Script | Purpose |
|-------|--------|---------|
| Planning DAG | `Build-PlanningDAG.ps1` | AI-powered advisory DAG generated before launch. Uses an architect agent to analyze the plan and suggest role assignments and dependencies. |
| Solid DAG | `Build-DependencyGraph.ps1` | Deterministic DAG built from actual room configs after war-rooms are created. This is the authoritative execution graph. |

Advisory dependencies from the planning DAG are merged only when the author
hasn't specified explicit `depends_on:` in the markdown.
