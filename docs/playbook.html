01 · Foundation

The Pyramid Principle

Borrowed from consulting, the Pyramid Principle (Barbara Minto, McKinsey) is the most important thinking structure for anyone writing prompts, EPICs, or technical specs for AI agents. The rule is simple: lead with your conclusion, then prove it.

CONCLUSION FIRST
KEY ARGUMENTS
SUPPORTING EVIDENCE
Bottom-Up thinking → Top-Down communication

Most engineers write context the way they think — building up from details to conclusions. This is the wrong order for agents. An LLM processes your prompt top-to-bottom and must hold all prior context in its attention window. If the conclusion is at the bottom, the agent has already made decisions before it knows what you want.

Applied to Prompts & Context Files

❌ Wrong order (bottom-up)

"There are several edge cases in the current payment processor. The retry logic is inconsistent, the error messages are not standardized, and the timeout handling differs across methods. Therefore, I want you to refactor the PaymentService class."

✅ Pyramid order (conclusion first)

"Refactor PaymentService to enforce consistent error handling. Three issues drive this: (1) inconsistent retry logic, (2) non-standard error messages, (3) timeout handling differs per method. Focus on these three changes only."

The Three Pyramid Rules

Rule Meaning In Practice
BLUF Bottom Line Up Front First sentence = the task/decision
MECE Mutually Exclusive, Collectively Exhaustive No overlap in acceptance criteria; no gaps
Rule of 3 Group support into ≤3 clusters Max 3 goals per EPIC phase, 3 AC per Story
The Pyramid Principle is the foundation of every other concept in this handbook. EPICs lead with the outcome. Prompts lead with the task. Context files lead with the current state. Master this, and everything else becomes cleaner.
02 · Foundation

Prompt-Driven Context

In classical software, your source code is the product. In agentic systems, your context is the source code. What you give the agent — its system prompt, its PLAN.md, its CONTEXT.md, its examples — fully determines what it builds.

Core principle

The quality of the AI's output is entirely a function of the quality of its input context. There is no "prompt magic." There is only context quality.

What "Context" Means in This System

📄

CONTEXT.md

The agent's memory. Tech stack, current phase, blockers, routing tags. Re-read at every session start.

🗺️

PLAN.md

The Supervisor's spec. Phase goals, tasks per role, AC, DoD. This is the source of truth for execution.

🔧

SKILL.md

Capability instruction files. Tell the agent how to do a class of tasks. Loaded on-demand via MCP.

📋

TODO.md

Engineer's append-only log. Release notes per version. QA reads this to understand what changed.

The Context Degradation Problem

As a conversation or task grows longer, context quality degrades. Agents start to drift, forget earlier constraints, or hallucinate decisions. The solution is not a better model — it is structured context reset. 

# At the start of EVERY agent session, run:
cat {feature}/CONTEXT.md           # Current state — phase, blockers, routing
cat {feature}/references/Supervisor/PLAN.md  # Active instructions
python room.py {feature} --since 0 | tail -20  # Recent events

# This is not optional. This IS the agent's working memory.

Prompt-Driven Context: The Three Layers

Layer File Who Writes When Read
System SKILL.md, ROLE.md Architect / Human Session init
Planning PLAN.md, CONTEXT.md Supervisor Agent Every task start
Execution TODO.md, QA.md Engineer / QA Agent Review & handoff
If you find yourself correcting an agent mid-task, you have a context problem, not a model problem. Go back and improve the PLAN.md before the next cycle.
03 · Foundation

Context Engineering

Context Engineering is the discipline of designing, structuring, and maintaining the information environment that agents operate in. Think of it as database schema design — but for an agent's mind.

A poorly-structured context produces hallucinations, out-of-scope changes, and broken contracts between agents. A well-structured context makes agents predictably capable.

The Four Properties of Good Context

Property Definition Anti-pattern
Bounded One context per feature. No cross-feature contamination. Monolithic PLAN.md covering 10 features
Current CONTEXT.md reflects actual current state, not intended state. Phase marked "Phase 1" when Phase 3 is running
Append-only History is never deleted. Agents read the full chain of decisions. Overwriting PLAN.md v1 with v2
Role-scoped Each agent owns exactly one file. Reads others' as read-only. Engineer editing QA.md

04 · Architecture

MCP & Skills

MCP (Model Context Protocol) is the open standard for connecting AI agents to tools and data sources. Rather than stuffing all capability into a system prompt, MCP lets you compose capabilities dynamically. Think of it as a package manager for agent tools.

Skills (SKILL.md files) are our layer on top of MCP. A Skill is a markdown file that instructs an agent how to use a class of tools or perform a domain task. Skills are loaded on-demand, keeping base context lean.

MCP Architecture in OSTwin

┌──────────────────────────────────────────────┐
│            Agent (Claude Code / Cursor)       │
│                                              │
│  Reads: CONTEXT.md → PLAN.md → SKILL.md     │
│         ↓                                    │
│  Calls: MCP servers (tools)                  │
│         ↓                                    │
│  Writes: TODO.md → ROOM.jsonl               │
└──────────────────────────────────────────────┘
         │             │              │
    [filesystem]  [database]    [external API]
      MCP server   MCP server     MCP server

Skill File Design Rules

  • Lead with trigger conditions — when to use this skill, when not to
  • Include a worked example — agents learn from demonstration, not just instruction
  • State constraints explicitly — what the skill must never do
  • Version your skills — changes to SKILL.md break running agents; version intentionally
  • Keep skills single-purpose — one SKILL.md per domain (docx, pdf, api, etc.)

MCP Server Scoping — Role-Based Access

Not every agent needs every tool. In OSTwin's architecture, MCP server access is scoped by role. This is a security boundary, not just convenience.

Role MCP Servers Allowed Rationale
Supervisor filesystem (read), ROOM bus Plans only; no code execution
Engineer filesystem (rw), shell, git, DB Full execution scope
QA filesystem (read), test runner, ROOM bus Validate only; no write to src
DevOps cloud CLI, registry, ROOM bus Deploy only; no src modification
05 · Architecture

Roles: Worker & Evaluator

The fundamental unit of agentic collaboration is the Worker / Evaluator pair. One agent produces; another validates. This maps directly to our Engineer / QA roles, but the principle generalizes across all multi-agent systems.

Core insight

A single agent cannot reliably evaluate its own output. The moment you separate production from evaluation into distinct agents with distinct context, quality dramatically improves. Self-review is a cognitive trap for both humans and LLMs.

Worker Agent (Engineer) Engineer

The Worker's job is to implement the specification in PLAN.md. It reads the Supervisor's instructions, claims a task, implements, writes release notes, and signals readiness.

  • CLAIM before starting any task
  • WORKING heartbeat every ~5 min on long tasks
  • Write TODO.md Release Notes FIRST, then post RELEASE event
  • Wait for QA response via --wait-for BUG,DELIVER
  • Max 3 BUG cycles, then post BLOCKED and escalate

Evaluator Agent (QA) QA

The Evaluator's job is to validate against the Acceptance Criteria in PLAN.md — not against its own judgment. QA does not add scope; it enforces the contract.

  • REVIEWING before starting review (prevents duplicate QA)
  • Read TODO.md Release Notes before testing
  • Write QA.md bug table FIRST, then post BUG event
  • Test against PLAN.md AC, not personal preference
  • Post Deliver Plan in QA.md before posting DELIVER

The Evaluation Contract

QA never invents requirements. Every bug filed must reference a specific Acceptance Criterion from PLAN.md. If it's not in the AC, it's not a bug — it's a new story. Scope creep kills velocity.

Autonomous Loop: Worker ↔ Evaluator

06 · Architecture

English as the Best Embedding Language

When writing PLAN.md, SKILL.md, acceptance criteria, or any context consumed by an LLM — write in English. This is not a cultural preference. It is a technical decision with measurable impact on agent output quality.

Why English Outperforms Other Languages for LLM Context

Factor English Vietnamese / Other
Training data density ~70–80% of most LLM pretraining corpora <2% for Vietnamese; other languages vary
Token efficiency 1 English word ≈ 1–1.5 tokens 1 Vietnamese word ≈ 2–4 tokens
Instruction following Highest benchmark scores across all models Accuracy degrades for precise technical instructions
Code alignment Code is written in English; context matches code Mismatch between context language and code language
RLHF alignment Most human feedback data is English-based RLHF coverage is sparse

The Token Budget Argument

LLMs have a finite context window. Every token spent on verbose or inefficient language is a token not available for code, logic, and reasoning. English maximizes the information-to-token ratio for all major models. 

# Vietnamese (original)
"Kiểm tra xem hàm xác thực người dùng có trả về lỗi 401 khi token hết hạn không"
# Token count: ~30 tokens

# English (equivalent)
"Verify auth function returns 401 when token is expired"
# Token count: ~11 tokens

# Same meaning. 3x more efficient. Higher model compliance.
The iGOT.AI Rule

All PLAN.md, CONTEXT.md, SKILL.md, AC, and DoD are written in English. User-facing content, documentation for clients, and product descriptions can be localized. Agent context is always English.

Ubiquitous Language in English

Domain Driven Design (see next section) requires a Ubiquitous Language — a shared vocabulary used by both humans and the system. If your domain model is in English, your agents understand it natively. Mixed-language contexts force the model to translate, introducing ambiguity at the most critical moment: understanding the spec.

07 · Architecture

Domain-Driven Design & TDD for Agents

Domain-Driven Design (DDD)

DDD is a software design approach that models the system around the business domain rather than technical constraints. In agentic systems, DDD shapes how we write EPICs, name entities, and define bounded contexts for agents.

📖

Ubiquitous Language

A shared vocabulary between engineers, agents, and domain experts. Every term in PLAN.md must match the code domain model precisely.

🏛️

Bounded Context

Each feature context folder = one bounded context. Agents operating in payment-flow/ don't cross into notifications/.

🧩

Aggregates

Group related entities that change together. A PaymentSession aggregate owns all state changes — agents modify it as a unit.

📣

Domain Events

ROOM.jsonl IS your event bus. RELEASE, BUG, DELIVER are domain events in the agent SDLC bounded context.

DDD Applied: Writing PLAN.md with Ubiquitous Language

## Phase 2 — Payment Flow

### Domain Model
# Use these exact names in code, comments, and tests:
PaymentSession   → represents one checkout attempt
PaymentIntent    → Stripe object; must mirror local state
SettlementRecord → immutable audit log entry post-capture

### Tasks [backend]
- Task 1: Implement PaymentSession.create() — idempotent, returns SessionId
- Task 2: Implement PaymentIntent.sync() — polls Stripe, updates local state
- Task 3: On capture, emit PaymentCaptured domain event → write SettlementRecord

When the Engineer reads this, the domain vocabulary is unambiguous. There is no guessing what "payment object" means. The agent generates code with the correct naming — which means your codebase stays consistent across agent sessions.

Test-Driven Development (TDD) for Agent Tasks

TDD inverts the workflow: write the test first, then write the code that passes it. For agentic systems, TDD maps directly to writing Acceptance Criteria before creating tasks in PLAN.md.

The Agentic TDD Loop

1. Supervisor writes AC (the "test") in PLAN.md  ·  2. Engineer implements until AC passes  ·  3. QA runs AC as literal test cases  ·  4. DELIVER only when all AC are green

Writing AC as Test Cases

## Acceptance Criteria — Task 1: PaymentSession.create()

# Format: GIVEN / WHEN / THEN
AC-1: GIVEN valid userId and cartId
      WHEN  PaymentSession.create() is called
      THEN  returns a unique SessionId AND persists to DB

AC-2: GIVEN the same userId + cartId is submitted twice
      WHEN  PaymentSession.create() is called
      THEN  returns the EXISTING SessionId (idempotent)

AC-3: GIVEN invalid userId (non-existent)
      WHEN  PaymentSession.create() is called
      THEN  throws UserNotFoundError (NOT a 500)

These AC are directly executable by QA. The agent running QA can write unit test stubs from them. There is no interpretation required. GIVEN/WHEN/THEN is the lingua franca of AC.

The EPIC Planning Standard

In agentic systems, the EPIC is not just a project management artifact — it is the primary source of truth for everything the agent will produce. A weak EPIC produces weak code. A precise EPIC produces precise code. Every field in the EPIC format exists for a reason: it gives the agent unambiguous instruction, gives QA testable criteria, and gives the Supervisor traceable accountability.

The EPIC is the spec. The EPIC is the test. The EPIC is the contract between human intent and machine execution.

The Canonical EPIC Format

Every EPIC in this system follows a single, rigid structure. This is not a suggestion — it is a standard. Agents, QA, and the Supervisor all parse this format explicitly. Deviation breaks the contract. 

## EPIC-001 - {Outcome Statement}
Roles:        @engineer, @qa, @devops
# One-paragraph definition of the business problem and the goal.
# Lead with the conclusion (Pyramid Principle).
# Then state what this EPIC delivers and why it matters.

### Definition of Done
- [ ] All Acceptance Criteria pass (QA verified)
- [ ] No HIGH or CRITICAL bugs open
- [ ] Unit test coverage ≥ 80% for new code
- [ ] CONTEXT.md updated to reflect completion
- [ ] Deployed to staging and smoke-tested

### Acceptance Criteria
- [ ] AC-1: GIVEN {precondition} WHEN {action} THEN {expected result}
- [ ] AC-2: GIVEN {precondition} WHEN {action} THEN {expected result}
- [ ] AC-3: GIVEN {edge case} WHEN {action} THEN {expected result}

### Tasks
- [ ] Task 1: {technical deliverable} — [routing-tag]
- [ ] Task 2: {technical deliverable} — [routing-tag]
- [ ] Task 3: {technical deliverable} — [routing-tag]

depends_on: [EPIC-000]

Field-by-Field Anatomy

Each field in the EPIC format has a specific purpose, a specific audience, and a specific failure mode when omitted. The table below is a reference for anyone writing or reviewing an EPIC.

Field Purpose Who Reads It If Missing
EPIC-{NNN} Unique, sequential identifier. Enables cross-referencing and dependency tracking. Everyone EPICs become untraceable; depends_on links break.
Outcome Statement One-line business result. Uses Pyramid Principle — conclusion first. Supervisor, Stakeholder Agent cannot determine when the EPIC is "done."
Role Which agent roles participate. Controls MCP scoping and routing. Agent Router, Supervisor Tasks get assigned to wrong agents or not at all.
Definition & Goal Business context paragraph. States problem, solution, and value. Engineer, QA Agent has no "why" — makes suboptimal design decisions.
Definition of Done Phase/EPIC-level exit gate. Binary checklist — all must pass. QA (as DELIVER gate) QA cannot determine when to post DELIVER; scope creep.
Acceptance Criteria Task-level test cases. GIVEN/WHEN/THEN format. QA executes these. QA (as BUG/PASS gate) QA invents criteria → scope drift and arbitrary bugs.
Tasks Technical deliverables with routing tags. Engineer claims and implements. Engineer, Supervisor No work items to claim. Agents idle or hallucinate scope.
depends_on Explicit ordering. This EPIC cannot start until listed EPICs are DELIVER'd. Supervisor, Scheduler Parallel EPICs conflict on shared resources or interfaces.

EPIC ID Convention

The ID follows the pattern EPIC-{NNN} where NNN is a zero-padded, monotonically increasing integer. IDs are never reused, even if an EPIC is cancelled.

🔢

Sequential

EPIC-001, EPIC-002, EPIC-003… The number reflects creation order, not priority. Priority is expressed in the depends_on graph.

🔗

Cross-Referenceable

Other EPICs, PLAN.md, CONTEXT.md, and ROOM events all reference EPICs by ID. This creates a traceable audit trail from intent to delivery.

🔒

Immutable

Once assigned, an EPIC ID never changes. If the scope fundamentally shifts, create a new EPIC with a new ID and link with depends_on.

Role Assignment

The Roles: field declares which agent roles are active participants in this EPIC. This is not informational — it controls MCP server scoping, task routing, and review responsibility.

Role Responsibility in EPIC MCP Access Badge
engineer Claims tasks, implements code, writes TODO.md release notes, posts RELEASE filesystem (rw), shell, git, DB Engineer
qa Reviews against AC, writes QA.md bugs, posts BUG or DELIVER filesystem (read), test runner, ROOM bus QA
supervisor Writes PLAN.md, defines AC & DoD, resolves BLOCKED escalations filesystem (read), ROOM bus Supervisor
devops Deploys to staging/production, posts DEPLOY event cloud CLI, registry, ROOM bus DevOps
Routing rule

Tasks tagged with [backend] or [frontend] are routed to Engineers with matching capabilities. Tasks tagged [infra] route to DevOps. The role list in the EPIC header determines which agent classes participate at all.

Definition & Goal Statement

The paragraph immediately following the header is the EPIC's north star. It must answer three questions in this order (applying the Pyramid Principle):

  • What is the outcome? — Lead with the conclusion. "Users can pay via Stripe."
  • Why does it matter? — Business justification. "Revenue cannot be collected without this."
  • What is the scope? — Explicit IN/OUT boundary. "IN: checkout flow. OUT: subscriptions, refunds."
❌ Wrong — detail-first, no conclusion

"We need to look at the current API response handling. There are several edge cases in how errors are propagated. The frontend sometimes sees 500s when it should see 400s. Various engineers have complained about inconsistent error shapes. Therefore, we should standardize the API error contract."

✅ Correct — conclusion first, scoped

"Standardize API error responses to a single, typed contract. Currently, error shapes are inconsistent across endpoints, causing frontend mishandling and poor UX. IN: all REST endpoints in /api/v2/. OUT: WebSocket events, gRPC services."

Definition of Done (DoD)

The DoD is the exit gate for a phase or EPIC. It is a binary checklist — every item must be true before QA posts the DELIVER event. DoD is written once per project by the Architect or Tech Lead, and applies uniformly to all EPICs. 

### Definition of Done
- [ ] All Acceptance Criteria pass (QA verified)
- [ ] No HIGH or CRITICAL bugs open in QA.md
- [ ] Unit test coverage ≥ 80% for all new code paths
- [ ] CONTEXT.md updated with completed phase and decisions
- [ ] TODO.md release notes written for every version shipped
- [ ] Deployed to staging environment and smoke-tested
- [ ] No regressions in existing test suites
DoD is not per-task

DoD applies to the entire phase or EPIC, not individual tasks. It answers: "Are we really done, or just done coding?" It includes process gates (tests, docs, deployment) that go beyond the AC.

Acceptance Criteria (AC)

AC are per-task test specifications written in GIVEN/WHEN/THEN format. They are the contract between the Supervisor's intent and the QA's verification. QA evaluates each AC item individually — if any fails, QA posts a BUG event referencing the specific AC ID. 

### Acceptance Criteria
- [ ] AC-1: GIVEN valid userId and cartId
            WHEN  PaymentSession.create() is called
            THEN  returns unique SessionId AND persists to DB

- [ ] AC-2: GIVEN same userId + cartId submitted twice
            WHEN  PaymentSession.create() is called
            THEN  returns EXISTING SessionId (idempotent)

- [ ] AC-3: GIVEN non-existent userId
            WHEN  PaymentSession.create() is called
            THEN  throws UserNotFoundError (NOT a 500)

The Three Rules of AC

Rule Why Anti-pattern
Always GIVEN/WHEN/THEN QA agents can parse this format directly into test stubs Prose AC like "system should handle errors properly"
Include edge cases Happy path alone misses 80% of bugs. State boundaries, nulls, duplicates. Only testing the golden path
Max 5 AC per task More than 5 signals the task should be split. Follows Rule of 3. 20 AC covering every micro-behavior

DoD vs AC — The Critical Difference

Acceptance Criteria (AC) Definition of Done (DoD)
Scope Per task / story Per phase / EPIC
Who writes Supervisor, before engineering Architect / Tech Lead, once per project
Who checks QA agent, against each AC item QA agent, as exit criteria for DELIVER
Format GIVEN / WHEN / THEN Checklist with binary pass/fail
Failure action Post BUG event, cycle back Block DELIVER, escalate to Supervisor
Analogy "Does this feature work correctly?" "Is this release shippable?"
The relationship

AC is a subset of DoD. All AC passing is one item on the DoD checklist. DoD adds process gates that are not functional tests: coverage thresholds, documentation, deployment, regression checks.

Tasks — Technical Deliverables

Tasks are the atomic work items claimed by Engineers. Each task has a routing tag, a clear deliverable, and maps to one or more AC. Tasks are claimed via the ROOM event bus before work begins. 

### Tasks
- [ ] Task 1: Implement PaymentSession.create() endpoint — [backend]
- [ ] Task 2: Build checkout UI with Stripe Elements — [frontend]
- [ ] Task 3: Add PaymentCaptured domain event + SettlementRecord — [backend]
- [ ] Task 4: Write integration tests for payment flow — [qa]

Task Design Rules

  • One routing tag per task — [backend], [frontend], [infra], [qa]. Tasks with multiple tags should be split.
  • Verb-first deliverables — "Implement…", "Build…", "Add…", "Write…". Not "Payment stuff" or "Handle errors".
  • Max 7 tasks per EPIC — More tasks signal the EPIC should be decomposed into sub-EPICs.
  • Each task maps to ≥1 AC — If a task has no AC, it either doesn't belong or the AC is missing.
  • Claim before coding — Post CLAIM event to ROOM before starting. Prevents duplicate work in multi-agent systems.

Dependency Graph — depends_on

The depends_on field creates an explicit ordering between EPICs. An EPIC cannot begin until all EPICs in its depends_on list have posted a DELIVER event. This prevents parallel conflicts on shared resources, interfaces, or database schemas. 

## EPIC-003 - MCP Skill Registry
depends_on: [EPIC-001, EPIC-002]
# ↑ Cannot start until auth (001) and DB schema (002) are both DELIVER'd

## EPIC-004 - Dashboard Analytics
depends_on: [EPIC-003]
# ↑ Depends on registry (003), which transitively depends on 001 + 002

## EPIC-005 - Notification System
depends_on: [EPIC-001]
# ↑ Only needs auth (001). Can run in parallel with 003 and 004.
Scheduling rule

The Supervisor reads the depends_on graph to determine which EPICs can run in parallel and which must be sequential. An empty depends_on: [] means the EPIC has no prerequisites and can start immediately.

Worked Example — A Complete EPIC

Below is a fully-specified EPIC following every convention in this standard. Use this as a template when writing your own. 

## EPIC-003 - Centralized MCP Skill Registry
Roles: @engineer, @qa, @devops

Build a centralized skill registry that enables agents to discover,
load, and version SKILL.md files dynamically via MCP. Currently,
skills are hardcoded in system prompts, causing context bloat and
version drift across agent sessions.

IN scope:  Skill CRUD API, version tagging, MCP server integration
OUT scope: Skill authoring UI, permission model, skill marketplace

### Definition of Done
- [ ] All Acceptance Criteria pass (QA verified)
- [ ] No HIGH or CRITICAL bugs open
- [ ] Unit test coverage ≥ 80% for registry module
- [ ] CONTEXT.md updated with registry architecture decisions
- [ ] API deployed to staging; agents can query skills via MCP
- [ ] No regressions in existing skill loading paths

### Acceptance Criteria
- [ ] AC-1: GIVEN a valid skill name and version
            WHEN  GET /api/skills/{name}?version={v} is called
            THEN  returns the SKILL.md content with 200
- [ ] AC-2: GIVEN a non-existent skill name
            WHEN  GET /api/skills/{name} is called
            THEN  returns SkillNotFoundError with 404
- [ ] AC-3: GIVEN a new SKILL.md file is registered
            WHEN  POST /api/skills is called with valid payload
            THEN  skill is persisted and queryable within 5 seconds
- [ ] AC-4: GIVEN an existing skill with version v1
            WHEN  a new version v2 is published
            THEN  both v1 and v2 are independently retrievable

### Tasks
- [ ] Task 1: Implement SkillRegistry service + Postgres schema — [backend]
- [ ] Task 2: Build REST endpoints: GET/POST/PUT /api/skills — [backend]
- [ ] Task 3: Implement MCP server adapter for skill discovery — [backend]
- [ ] Task 4: Write integration tests for all AC — [qa]
- [ ] Task 5: Deploy registry service to staging — [infra]

depends_on: [EPIC-001, EPIC-002]

Common EPIC Anti-Patterns

❌ Anti-patterns to avoid

Vague goals: "Improve the dashboard" — agents cannot test completion.
No scope boundary: EPIC grows during execution — agents hallucinate new features.
Missing AC: Engineer ships, QA has no test criteria — DELIVER is arbitrary.
AC written after engineering: The spec matches the implementation, not the intent.
No depends_on: Parallel EPICs modify the same DB schema, causing conflicts.
Missing roles: Tasks routed to agents that lack MCP access to execute them.
Prose DoD: "The feature should work well" — not testable, not binary.

EPIC Quality Checklist

Before submitting any EPIC, verify all of the following:

  • Has a unique EPIC-{NNN} ID — sequential, zero-padded, never reused
  • Outcome statement is one sentence — Pyramid Principle BLUF
  • Roles are explicitly listed — Role: engineer, qa
  • Definition paragraph states problem + solution + scope
  • IN/OUT scope is explicit — eliminates scope creep
  • DoD is a binary checklist (≥5 items) — pass/fail, not "good enough"
  • Every task has ≥2 AC in GIVEN/WHEN/THEN format
  • Tasks have routing tags — [backend], [frontend], [infra]
  • depends_on is declared — even if empty []
  • All content is in English — token-efficient, highest model compliance
09 · Execution

Agentic Ops: Best Practices

Agentic Ops is the operational discipline of running multi-agent systems reliably in production. The key insight: agents fail differently than humans. They don't get tired — they hallucinate silently, get stuck in loops, or produce plausible-looking but wrong output. Your ops practices must account for these failure modes.

The Five Pillars of Agentic Ops

📡

Observable

Every action is posted to ROOM.jsonl. If it's not in the event log, it didn't happen operationally.

🔒

Bounded

Agents operate within scoped MCP servers. No agent has more access than its role requires.

🔄

Recoverable

All context files are append-only. Any state can be replayed. No destructive operations on context.

⏱️

Heartbeated

Long-running agents post WORKING events every ~5 min. Waiting agents know the difference between a slow agent and a dead one.

📊

Escalating

Max 3 BUG cycles. On cycle 4, post BLOCKED and escalate to Supervisor. No infinite loops.

The File-First Rule

Critical operational rule

Always write your file completely before posting the ROOM event. The ROOM event is a pointer, not a payload. Posting RELEASE before writing TODO.md means QA has nothing to read. Posting BUG before writing QA.md means the Engineer has no bugs to fix. File first. Event second. Always.

Multi-Agent Coordination — Preventing Duplicate Work

# Step 1: Check what's already claimed
python room.py {feature} --claimed-tasks

# Step 2: Claim YOUR task BEFORE starting
python room.py {feature} engineer-1 CLAIM "Phase 2" "" "Task 3: build auth middleware"
# Other Engineers see this CLAIM and will NOT pick up Task 3

# Step 3: Heartbeat every ~5 min during long work
python room.py {feature} engineer-1 WORKING "Phase 2" "" "implementing auth middleware — 60% done"
# QA sees WORKING → knows Engineer is alive → does not time out

# Step 4: Write file. Post event.
# TODO.md release notes written first ↑
python room.py {feature} engineer-1 RELEASE "Phase 2" v2 "Release v2 ready — see TODO.md"

ROOM.jsonl Event Reference

Event Type Sender Meaning
PLAN Terminal Supervisor New phase plan posted
CLAIM Heartbeat Engineer "I'm taking this task"
WORKING Heartbeat Engineer / QA "Still alive, in progress"
RELEASE Terminal Engineer "Implementation ready for QA"
REVIEWING Heartbeat QA "I'm reviewing this release"
BUG Terminal QA "Bugs found — see QA.md"
FIX Terminal Engineer "Bugs fixed — see TODO.md"
DELIVER Terminal QA "Phase cleared — ready for next"
BLOCKED Terminal Any "Escalate to Supervisor"
DEPLOY Terminal DevOps "Deployed to target environment"

10 · The Core Thesis

Code Quality = Context Quality

Everything in this handbook converges on a single, non-negotiable truth:

The quality of the AI's code is entirely determined by the quality of the EPICs, Definitions of Done, and Acceptance Criteria.

Not the model version. Not the IDE. Not the agent framework. The context.

The Quality Chain

EPIC clarity
Business outcome defined
→
Precise AC
GIVEN/WHEN/THEN
→

Agent output

Introducing ambiguity at any link in this chain propagates forward and amplifies. A vague EPIC produces vague PLAN.md. A vague PLAN.md produces inconsistent implementation. Inconsistent implementation produces meaningless QA. And meaningless QA produces a broken product.

The Diagnostic: Where Did Quality Break?

When agent output is poor, diagnose the context chain — not the agent:

Symptom Root Cause Fix
Agent implements the wrong thing EPIC goal was ambiguous or missing Rewrite EPIC with Pyramid Principle BLUF
Agent adds unrequested features No explicit scope boundary in EPIC Add "OUT OF SCOPE" section to PLAN.md
QA can't verify the release AC written in prose, not GIVEN/WHEN/THEN Rewrite AC as executable test cases
Agent breaks things in Phase 3 that worked in Phase 1 CONTEXT.md not updated; agent lacks history Enforce CONTEXT.md update after every DELIVER
Different agents use different variable names No Ubiquitous Language in PLAN.md Add domain model glossary; enforce in AC
Agent loops forever without progress No escalation rule; no BLOCKED ceiling Enforce max 3 BUG cycles + BLOCKED protocol

The Multiplier Effect

Poor human planning multiplied by autonomous agents produces poor output at scale, and fast. An agent that runs 22 hours while you sleep will generate 22 hours of wrong code if given a bad EPIC. This is not a model failure — it is a planning failure wearing a technology mask.

Conversely, a precise EPIC — 2 hours of careful human planning — can produce 22 hours of correct autonomous execution. This is the 2-22 Framework: invest 2 hours in context quality, unlock 22 hours of agent productivity.

The 2-22 Framework

2 hours: Human writes precise EPIC, AC, DoD, CONTEXT.md, seeds the ROOM.
22 hours: Agents execute, loop, validate, and ship — autonomously, correctly.

The Engineer's Checklist Before Any EPIC

  • EPIC has a single, testable outcome statement (Pyramid BLUF)
  • Scope boundary is explicit — what is IN and what is OUT
  • Every task has ≥2 AC in GIVEN/WHEN/THEN format
  • DoD checklist is written and binary (pass/fail, not "good enough")
  • Domain model entities are named (DDD Ubiquitous Language)
  • CONTEXT.md reflects the current actual state, not intended state
  • Routing tags assign tasks to the correct agents
  • All context files are in English
  • Max phases ≤ 3 per EPIC; escalate larger work to multiple EPICs
  • Escalation ceiling defined — max BUG cycles and BLOCKED trigger
The Final Word

Every technique in this handbook — the Pyramid Principle, Prompt-Driven Context, DDD, TDD, MCP scoping, the Worker/Evaluator split, English as the embedding language — serves one purpose: to make your context so precise that the agent cannot fail to understand what you want. That is the craft of agentic engineering.
