# PR: Unified AI Gateway, Centralized Memory Storage, Memory Lifecycle Management

**Branch:** `refactor/unify-llm-provider`
**Base:** `main`
**Files changed:** 41 (+2,558 / -2,089)

---

## Summary

Three major improvements to the memory system, building on the HTTP memory transport (PR #106):

1. **Unified AI Gateway** — All memory LLM/embedding calls now route through `dashboard/ai/`. Deleted 427 lines of redundant LLM controller code. Every AI call is monitorable.
2. **Centralized Memory Storage** — Memories stored at `~/.ostwin/memory/<plan_id>/` with project symlinks. Stable hash-based plan IDs prevent namespace collisions.
3. **Memory Lifecycle Management** — Clear, archive, export, delete memories from CLI, REST API, and dashboard UI.

---

## What changed

### 1. Unified AI Gateway

**Problem:** The memory system had its own LLM calling stack (`llm_controller.py` — 6 controller classes, litellm direct calls) completely separate from the dashboard's AI gateway (`dashboard/ai/`). No monitoring, no centralized config.

**Solution:** Route all memory AI calls through `dashboard/ai/`:
- `memory_system.py` — 3 LLM call sites now use `dashboard.ai.get_completion(purpose='memory')` via injected `completion_fn`
- `retrievers.py` — embeddings now use `dashboard.ai.get_embedding()` via injected `embed_fn`
- Deleted `llm_controller.py` (427 lines, 6 controller classes)
- Removed litellm, sentence-transformers, transformers from memory system imports
- Every AI call visible in AI Monitor dashboard panel

**UI changes:**
- Restored `AIMonitorPanel.tsx` — shows live call stats, latency, per-model/per-purpose breakdowns
- Rewrote `MemoryPanel.tsx` — replaced obsolete LLM/embedding backend pickers with AI gateway status card
- Added `AISettings` model + `'ai'` namespace to settings system
- Wired `/api/ai` routes (complete, embed, stats) into `api.py`
- Every `get_embedding()` call now records model, text count, latency to the monitor

### 2. Centralized Memory Storage

**Problem:** Memory URL template used `{env:PROJECT_DIR}` which couldn't be resolved in the global opencode.json. Per-project `.memory/` directories scattered across working dirs.

**Solution:**
- `mcp-builtin.json` uses bare URL: `http://localhost:3366/api/memory-pool/mcp` (no query params)
- `init.ps1` patches `?plan_id=<hash>` at runtime and creates symlink: `project/.memory → ~/.ostwin/memory/<plan_id>/`
- `Start-Plan.ps1` and `ostwin` binary generate stable hash plan IDs: `sha256(working_dir:filename)[:12]`
- `memory_mcp.py` resolves `plan_id` to centralized path
- Removed `persist_dir` query param support — `plan_id` only

**Storage layout:**
```
~/.ostwin/memory/
├── _global/              ← no plan context
├── d6aff5dcdb4e/         ← plan-specific (hash ID)
│   ├── notes/
│   └── vectordb/

~/ostwin-workingdir/gold-mining/
└── .memory → ~/.ostwin/memory/d6aff5dcdb4e/   ← symlink
```

### 3. Memory Lifecycle Management

**Problem:** Memories persist forever. No way to clear, archive, export, or delete — from CLI, API, or UI.

**Backend:**
- `AgenticMemorySystem.clear()` — wipes all notes + vector index + files
- `delete_memory` MCP tool exposed via HTTP pool

**REST API (5 new endpoints in amem.py):**

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/api/amem/namespaces` | List all namespaces with stats, disk usage, plan title |
| DELETE | `/api/amem/{plan_id}` | Clear all notes for a plan |
| DELETE | `/api/amem/{plan_id}/notes/{note_id}` | Delete a single note |
| POST | `/api/amem/{plan_id}/archive` | Archive → timestamped backup, start fresh |
| GET | `/api/amem/{plan_id}/export` | Download as .tar.gz |

**CLI (`ostwin memory` subcommand):**
```
ostwin memory list                      List all namespaces with stats
ostwin memory stats <plan_id>           Show stats for a namespace
ostwin memory tree <plan_id>            Show note directory tree
ostwin memory clear <plan_id> --force   Clear all notes
ostwin memory delete <plan_id> <id>     Delete a single note
ostwin memory archive <plan_id>         Archive and start fresh
ostwin memory export <plan_id>          Export as .tar.gz
```

**Dashboard UI:**
- MemoryTab header: Archive / Clear / Export buttons
- Each note: hover to reveal delete icon (with confirmation)

## Migration notes

- Run `install.sh` to propagate changes. The `merge_mcp_builtin.py` fix ensures the memory server type changes from `local` (stdio) to `remote` (HTTP) automatically.
- Existing per-project `.memory/` directories can be migrated with `bash scripts/migrate-memory.sh`.
- The dashboard must be running for memory to work (HTTP transport). Start with `ostwin start` or `bash ~/.ostwin/.agents/dashboard.sh`.
- `ostwin memory` CLI commands require the dashboard to be running (they call the REST API).
